## Design Summary

改进合同模板管理功能，解决用户提出的四个核心问题：
1. 模板删除功能缺失
2. 无法添加自定义映射字段
3. 字段添加到 JSON 后还需手动启用的交互冗余
4. 字段用途不透明，缺乏显示标签

## Alternatives Considered

### 方案 A：增量改进（推荐）
- **做法**：基于现有代码逐步添加缺失功能
  - 添加 DELETE /api/templates/:id 端点，带外键检查
  - 前端添加"添加自定义字段"按钮，支持字段名+标签
  - 简化启用逻辑：点击标签自动加入 activeSet，开关仅控制校验参与
  - 映射格式扩展：`{"fieldName": "显示标签"}`，UI 同时展示占位符和标签
- **优点**：改动可控，与现有架构兼容，快速交付
- **缺点**：不改变现有数据模型（fieldMap 仍是 JSON 对象）
- **为何采用**：用户需求明确，增量改进风险最低，可立即解决痛点

### 方案 B：全新字段配置 UI
- **做法**：用表格形式管理字段（字段名、标签、是否启用、排序），彻底替代 JSON 编辑器
- **优点**：更直观，无需手写 JSON
- **缺点**：开发量大，需迁移现有数据
- **为何未采用**：JSON 编辑器提供了灵活性，当前阶段不值得完全重写

### 方案 C：完全由后端驱动字段定义
- **做法**：后端定义所有可用字段（含系统字段+自定义字段），前端动态渲染
- **优点**：前后端分离更彻底，字段元数据可复用
- **缺点**：引入新的字段注册表，过度设计
- **为何未采用**：当前只有模板管理需要此功能，不值得抽象

## Agreed Approach

采用方案 A（增量改进），按以下顺序实现：
1. **模板删除**：后端删除前检查是否被合同引用，若引用则返回 409；前端添加删除按钮
2. **自定义字段**：添加按钮弹出表单收集字段名和标签，新字段自动追加到预设列表
3. **简化启用逻辑**：点击预设字段标签或添加自定义字段时，自动将该字段加入 activeSet 并保存
4. **显示标签**：fieldMap 中存储标签值，UI 展示 `${fieldName} (标签)`

## Key Decisions

- **删除策略**：使用硬删除（物理删除），但被合同使用的模板禁止删除（返回错误信息）。暂不实现软删除，因为模板一旦被使用，删除会破坏历史合同关联。
- **自定义字段存储**：直接写入 fieldMap JSON，格式与其他字段一致。不新增数据表。
- **启用状态默认值**：新添加的字段默认启用（加入 activeSet）。
- **UI 标签显示**：字段标签默认为字段名本身，用户可编辑 JSON 修改。

## Open Questions

- 是否需要支持编辑已有字段的标签？（当前用户可手动编辑 JSON 实现）
- 模板删除后，关联合同的 templateId 是否置空？（建议置空，并保留合同数据）
