## Context

社区/村委商铺租赁管理目前完全依赖纸质记录或 Excel，工作人员需要手动跟踪谁交了租、谁欠了费、什么时候该催。这个过程繁琐、易错、不可追溯。需要一个工具在后台自动完成记录、计算和提醒，同时不改变人的工作习惯。

系统以「合同」为核心锚点，第一个切入场景是商铺租赁，但架构预留了多资产类型扩展（车位、摊位、设备等）。

技术约束：
- 村委网络条件参差不齐，需支持完全离线单机运行
- 需要可升级到多人在线模式
- 打印输出要求尺寸精确（合同、三联收据）
- 非技术人员操作，UI 需极简直观

## Goals / Non-Goals

**Goals:**
- 一个完整的合同-收款-催缴闭环管理系统
- Go 后端分层架构，核心计算逻辑纯函数、100% 可单测
- Vue 3 前端 + Apple 设计系统，精致高级
- SQLite 本地模式（零依赖单机运行）和 PostgreSQL 服务端模式（多人同时访问）
- 后端 PDF 生成（合同、身份证复印件、三联收据）
- 合同模板上传 + 占位符字段映射
- 五级催缴清单自动生成，分级附带建议动作

**Non-Goals:**
- OCR 身份证识别（预留接口，本次不实现）
- 移动端 App（仅 Web 端）
- 在线支付集成
- 财务系统对接
- 多语言国际化
- 合同模板在线编辑（用户上传 .docx，系统做占位符替换）

## Decisions

### 1. 后端架构：Go 分层架构

**决策**：cmd → transport → service → domain ← repository

**理由**：
- domain 不依赖任何外层，核心计算逻辑零耦合，可独立单测
- repository 接口定义在 domain，SQLite/PostgreSQL 在 repository/ 下分别实现，启动时根据 config 注入
- service 层纯 Go 逻辑，不依赖 HTTP 框架，可脱离 Gin 测试

### 2. 前端架构：Vue 3 SPA 内嵌于 Go 二进制

**决策**：Vue 3 (Composition API + Pinia + Vue Router) 构建后由 Go embed.FS 内嵌

**理由**：
- 单一二进制分发，本地模式零依赖
- Apple 设计系统：毛玻璃卡片 (rgba(255,255,255,0.72) + backdrop-filter)、SF 色系 (#F5F5F7 背景、#007AFF 主色)、8px 基准网格
- 10 个页面视图，三大入口（签新合同、收租金、催缴清单）为核心交互

### 3. 数据库：Repository 接口 + 双实现

**决策**：Repository 接口定义在 domain 层，sqlite/ 和 postgres/ 分别实现

**理由**：
- SQLite 本地模式零依赖，数据文件可拷贝备份
- PostgreSQL 支持多人多地同时访问
- 迁移工具支持从 SQLite 升级到 PostgreSQL

### 4. 核心计算引擎：纯函数

**决策**：domain/calc/ 下实现 ContractCalc 和 ArrearsCalc，纯函数无副作用

**关键算法**：
- 应收总额：整月数 × 月租金 + 剩余天数 × (月租金/30)，结果可人工微调
- 钱用到哪天：整月按日历月（自动适应 28/29/30/31 天），余数按日租金计算
- 欠费判断：TotalReceived < TotalReceivable，单一标准
- 五级分级：按条件取最高匹配等级，一个合同只出现在一张清单上

### 5. 打印模块：后端 PDF 生成

**决策**：所有打印在后端完成，返回 PDF 二进制流

**理由**：
- 前端打印无法保证尺寸精度（三联收据需精确三等分）
- 合同替换占位符后渲染为 PDF，保证跨平台一致性
- 三联收据按本管理（ReceiptBook），打印时自动取号递增

### 6. 认证与授权

**决策**：JWT 认证，admin/operator 双角色

- admin：可管理用户
- operator：日常操作（签合同、收租金、看催缴）

### 7. 合同模板

**决策**：用户上传 .docx，定义占位符 → 系统字段映射，存储为 JSON

**理由**：
- 用户自行维护合同模板，系统不预设格式
- 字段映射灵活，适配不同地区的合同模板差异

### 8. ExtraFields 扩展机制

**决策**：资产和租户的非核心字段通过 ExtraFields (JSON) 存储，系统设置中定义扩展字段列表，UI 根据配置动态渲染输入控件

**理由**：
- 核心计算仅依赖租期和租金，其他字段为辅助信息
- 用户可按需自定义（如租户增加「职业」「紧急联系人」），无需改表

## Risks / Trade-offs

- [风险] SQLite 并发写入瓶颈 → **缓解**：本地模式为单机使用，并发低；高并发场景使用 PostgreSQL 模式
- [风险] "钱用到哪天"日历月算法在不同月份长度下可能有 1 天误差 → **缓解**：日租金使用月租金/30 统一基准，避免日租金浮动；结果与人工判断一致即可
- [风险] Vue SPA 首次加载较慢 → **缓解**：内嵌模式本地访问，延迟可忽略；服务端模式配置 Nginx gzip 压缩
- [风险] .docx 模板占位符替换兼容性 → **缓解**：使用成熟 Go 库处理 docx，提供模板预览和字段映射验证
- [风险] 五级催缴条件在边界日期可能出现歧义 → **缓解**：催缴引擎使用服务端日期，每日自动刷新，边界清晰
